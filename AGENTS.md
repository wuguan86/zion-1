# AGENTS.md

## 项目说明

Zion Admin 是一个前后端分离的后台管理系统。后端采用 Maven 多模块结构，主要技术栈包含 Java、Spring Boot、MyBatis-Plus、Sa-Token、Redis、MySQL 等；前端包含 Vue 3、Vite、TypeScript、Naive UI、Pinia、Vue Router、Axios、ECharts 等。

DK 25 (LTS), Spring Boot 4.x, MyBatis-Plus

## 适用范围

本文件适用于仓库根目录及所有子目录。所有自动化代理、代码助手和协作者在本项目中进行分析、编码、测试、文档编写和命令输出时，都必须遵守本文件规则。

## 1. 通用原则

- **JDK 25 特性**: 优先使用“虚拟线程 (Virtual Threads)”处理并发，利用“记录模式 (Record Patterns)”和“Switch 模式匹配”简化代码。
- 保持代码简洁，遵循 KISS 原则，避免过度设计和不必要的抽象。
- 保持模块化设计，按照已有模块边界和目录结构扩展功能。
- 针对动态字段优先使用 `JSONB` 类型。字段命名使用小写加下划线（snake_case）。
- PC端业务相关的表统一用“web_”开头，并且id使用雪花算法
- 所有变量名、函数名、类名、方法名必须具有描述性，使用英文命名，禁止使用拼音命名。
- 在修改代码前，必须先分析现有逻辑、调用链、数据结构和边界条件，确保新代码不破坏原有功能。
- 注释必须使用中文，并且要详细说明业务意图、关键条件和复杂逻辑；不要为显而易见的代码添加空泛注释。
- 需要日志时必须添加必要日志，日志内容必须使用中文，并清晰说明操作、结果和异常原因。
- 与用户沟通、任务说明、总结、测试结果和错误说明必须使用中文输出。
- 遵循单一职责原则：单个文件原则上不超过 500 行；如果超过 500 行，必须优先拆分为子组件、工具类、服务类或独立模块。

## 2. 修改代码前的要求

- 优先阅读现有实现，确认项目已有的命名风格、分层方式、异常处理、日志方式和测试方式。
- 不随意引入新的框架、依赖或架构模式；确有必要时，需要说明原因并保持影响范围最小。
- 不修改与当前任务无关的代码、格式、依赖版本或配置。
- 遇到已有未提交改动时，必须视为用户改动，不得回退、覆盖或删除，除非用户明确要求。

## 3. 后端开发规范

- 遵循现有 Maven 多模块边界，公共能力放在合适的 common、infra、core、api、job 或 starter 模块中。
- Controller 只负责请求接收、参数校验和响应组织，业务逻辑应放入 Service 或领域相关组件。
- 异常处理、返回结构、权限校验和分页查询应优先复用项目已有封装。
- 日志应包含关键业务标识和异常上下文，避免输出敏感信息。
- 新增或修改复杂逻辑时，应补充必要的单元测试或集成验证。

## 4. 前端开发规范

- Vue 相关代码优先使用 Vue 3 Composition API、`<script setup>` 和 TypeScript。
- 页面、组件、状态管理、路由和 API 请求应遵循现有目录结构与封装方式。
- 组件应保持单一职责；当页面或组件过大时，拆分为子组件、组合式函数或工具方法。
- 用户可见文案、提示、错误信息和日志输出应使用中文。
- UI 修改应与现有 Naive UI 风格保持一致，避免引入不协调的视觉样式。

## 5. 验证与交付

- 完成修改后，根据变更范围运行必要的构建、测试、类型检查或 lint 命令。
- 如果无法运行验证命令，必须在最终说明中明确原因和风险。
- 最终回复应简洁说明完成内容、涉及文件和验证结果。
- 不得声称“已完成”“已修复”或“测试通过”，除非已经通过相应验证。
