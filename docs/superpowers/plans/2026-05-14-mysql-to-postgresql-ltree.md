**Goal:** 将 Zion-Admin 从 MySQL 8.0 迁移到 PostgreSQL，并在 `sys_dept` 表上使用 `ltree` 扩展替代 `ancestors` 物化路径。

**Tech Stack:** Spring Boot 4.0.6, MyBatis-Plus 3.5.15, Sa-Token 1.45.0, Druid 1.2.28, PostgreSQL JDBC Driver 42.7.x

```java
@Bean
public MybatisPlusInterceptor mybatisPlusInterceptor(DataPermissionInterceptor dataPermissionInterceptor) {
    MybatisPlusInterceptor interceptor = new MybatisPlusInterceptor();
    // 1. 数据权限插件（先执行 SQL 包装）
    interceptor.addInnerInterceptor(dataPermissionInterceptor);
    // 2. 分页插件（最后执行，确保 LIMIT 在最外层）
    interceptor.addInnerInterceptor(new PaginationInnerInterceptor(DbType.POSTGRE_SQL));
    return interceptor;
}
```

MyBatis-Plus 会自动将分页从 MySQL 的 `LIMIT ?, ?` 转为 PostgreSQL 的 `LIMIT ? OFFSET ?`。`IdType.AUTO` 会自动适配 PG 的 `SERIAL`/`BIGSERIAL`。

- [ ] **Step 2: 转换规则速查表**

| MySQL | PostgreSQL |
|-------|-----------|
| `` `table_name` `` | `table_name` (无需引号) |
| `bigint NOT NULL AUTO_INCREMENT` | `bigint GENERATED BY DEFAULT AS IDENTITY` |
| `datetime` | `timestamp` |
| `tinyint` | `smallint` |
| `int` | `integer` |
| `varchar(N) CHARACTER SET utf8mb4 COLLATE utf8mb4_0900_ai_ci` | `varchar(N)` |
| `ENGINE=InnoDB` / `CHARACTER SET` / `COLLATE` / `ROW_FORMAT` | 全部删除 |
| `DEFAULT CURRENT_TIMESTAMP` | `DEFAULT CURRENT_TIMESTAMP` (兼容) |
| `ON UPDATE CURRENT_TIMESTAMP` | 删除（用应用层 `updateTime` 自动填充替代） |
| `` `backtick` `` | 直接删除反引号 |
| `COMMENT 'xxx'` | 保持不变 (PG 支持) |
| `INSERT INTO ... VALUES ...` | 大部分兼容，日期值 `'2026-01-29 22:42:08'` 改为 `'2026-01-29 22:42:08'::timestamp` |

- [ ] **Step 3: 特殊处理 sys_dept 表 — ltree 替代 ancestors**

原 MySQL 表：
```sql
CREATE TABLE `sys_dept` (
    `id` bigint NOT NULL AUTO_INCREMENT COMMENT '主键ID',
    `parent_id` bigint NULL DEFAULT 0 COMMENT '父部门ID',
    `ancestors` varchar(500) CHARACTER SET utf8mb4 COLLATE utf8mb4_0900_ai_ci NULL DEFAULT '' COMMENT '祖级列表',
    ...
```

新 PostgreSQL 表：
```sql
CREATE TABLE sys_dept (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    parent_id bigint DEFAULT 0,
    path ltree COMMENT '层级路径(ltree)',
    dept_name varchar(50) NOT NULL,
    sort integer DEFAULT 0,
    leader varchar(50),
    phone varchar(20),
    email varchar(100),
    status smallint DEFAULT 1,
    create_time timestamp DEFAULT CURRENT_TIMESTAMP,
    update_time timestamp DEFAULT CURRENT_TIMESTAMP,
    create_by bigint,
    update_by bigint,
    deleted smallint DEFAULT 0
);

-- ltree 索引，加速 <@ 和 @> 查询
CREATE INDEX idx_sys_dept_path ON sys_dept USING GIST (path);
```

### Task 6: 修改 SysDeptServiceImpl — 使用 ltree 路径逻辑

**Files:**

- [ ] **Step 1: 重写 create() 方法**

原代码使用 `ancestors` 逗号拼接，新代码用 `path` 句点拼接。因为 id 是 insert 后生成的，需要两步：先 save 获取 id，再 update path。

```java
@Override
public void create(SysDept dept) {
    if (dept.getParentId() == null || dept.getParentId() == 0) {
        dept.setParentId(0L);
        dept.setPath(null);
        this.save(dept);
        dept.setPath("0." + dept.getId());
    } else {
        SysDept parent = this.getById(dept.getParentId());
        if (parent == null) {
            throw new BusinessException("父部门不存在");
        }
        dept.setPath(null);
        this.save(dept);
        dept.setPath(parent.getPath() + "." + dept.getId());
    }
    this.updateById(dept);
}
```

- [ ] **Step 2: 重写 update() 方法**

当 `parentId` 改变时重新计算 `path`：

```java
@Override
public void update(SysDept dept) {
    SysDept existDept = this.getById(dept.getId());
    if (existDept == null) {
        throw new BusinessException("部门不存在");
    }
    if (dept.getParentId() != null && dept.getParentId().equals(dept.getId())) {
        throw new BusinessException("上级部门不能选择自己");
    }
    // 更新层级路径
    if (dept.getParentId() != null && !dept.getParentId().equals(existDept.getParentId())) {
        if (dept.getParentId() == 0) {
            dept.setPath("0." + dept.getId());
        } else {
            SysDept parent = this.getById(dept.getParentId());
            if (parent == null) {
                throw new BusinessException("父部门不存在");
            }
            dept.setPath(parent.getPath() + "." + dept.getId());
        }
    }
    this.updateById(dept);
}
```

- [ ] **Step 3: 重写 move() 方法**

```java
@Override
@Transactional(rollbackFor = Exception.class)
public void move(Long id, Long parentId, Integer sort) {
    SysDept dept = this.getById(id);
    if (dept == null) {
        throw new BusinessException("部门不存在");
    }
    if (id.equals(parentId)) {
        throw new BusinessException("上级部门不能选择自己");
    }

    dept.setParentId(parentId);
    if (sort != null) {
        dept.setSort(sort);
    }

    // 更新路径
    if (parentId == 0) {
        dept.setPath("0." + id);
    } else {
        SysDept parent = this.getById(parentId);
        if (parent == null) {
            throw new BusinessException("父部门不存在");
        }
        dept.setPath(parent.getPath() + "." + id);
    }

    this.updateById(dept);

    // 递归更新子部门的 path
    updateChildPaths(dept);
}
```

- [ ] **Step 4: 重写 updateChildAncestors() → updateChildPaths()**

原方法名 `updateChildAncestors`，用逗号拼接 ancestors。新方法用 ltree 路径：

```java
/**
 * 递归更新子部门的 ltree 路径
 */
private void updateChildPaths(SysDept parentDept) {
    List<SysDept> children = this.list(
        new LambdaQueryWrapper<SysDept>().eq(SysDept::getParentId, parentDept.getId()));
    for (SysDept child : children) {
        child.setPath(parentDept.getPath() + "." + child.getId());
        this.updateById(child);
        updateChildPaths(child);
    }
}
```

### Task 7: 修改 DataPermissionInterceptor — FIND_IN_SET → ltree

**Files:**
- Modify: `E:\project\Zion-Admin\zion-infra\zion-db\src\main\java\com\zion\db\interceptor\DataPermissionInterceptor.java`

这是本次迁移的**关键改动**。唯一使用 MySQL 特有 `FIND_IN_SET` 的地方。

- [ ] **Step 1: 修改 buildDataScopeFilter 中 scope==4 的 SQL**

原代码（第 134 行）：
```java
} else if (scope == 4 && userDeptId != null) {
    conditions.add(deptAlias + " IN (SELECT id FROM sys_dept WHERE id = " + userDeptId + " OR FIND_IN_SET(" + userDeptId + ", ancestors))");
}
```

替换为 ltree `<@` 操作符（"是后代或自身"）：

```java
} else if (scope == 4 && userDeptId != null) {
    conditions.add(deptAlias + " IN (SELECT id FROM sys_dept WHERE path <@ (SELECT path FROM sys_dept WHERE id = " + userDeptId + "))");
}
```

**语义解释：**
- `path <@ subtree_path` → 当前部门的 `path` 是目标路径的**后代或相等**
- 如果用户属于部门 2（path=`0.1.2`），子查询找到 `0.1.2`，外层 `<@` 操作符匹配所有以 `0.1.2` 为前缀的 path：`0.1.2`（自身）、`0.1.2.6`（子部门）、`0.1.2.6.10`（孙部门）

## 风险点与注意事项

### 1. Druid Wall Filter 兼容性
Druid 的 WallFilter 默认对 PG 支持较好，但如果遇到 SQL 被拦截，在 `application-dev.yml` 中暂时关闭 wall filter：
```yaml
filter:
  wall:
    enabled: false  # 暂时关闭，待确认 SQL 稳定性后重新开启
```

### 2. 分页查询差异
MyBatis-Plus 会自动适配分页。但如果有自定义的分页参数绑定（如直接在 SQL 中写 `LIMIT`），需要手动改为 PG 语法。当前项目没有这种情况。

### 3. ltree 路径长度限制
ltree 单个 label 最长 256 字符，总路径最长 65536 字符。部门 id 为 bigint，label 只会是数字，远超实际需求。

### 4. 雪花 ID 的替代方案
当前使用数据库自增 ID（`IdType.AUTO`）。如果未来需要切换到雪花 ID（`IdType.ASSIGN_ID`），部门创建时可以避免"先 save 再 update path"的两步操作，因为 id 在插入前就已确定。

### 5. sys_menu 不改 ltree
`sys_menu` 的 `parent_id` 树维持不变。ltree 的收益在于**频繁的"查子树"数据库查询**，而菜单树在应用启动时一次性加载到缓存，不在 DB 做递归查询。

### 6. 生产环境 deployment
在生产环境部署时，需要：
1. 备份 MySQL 数据
2. 使用数据迁移工具（如 pgloader 或自定义 ETL 脚本）迁移数据
3. 所有 datetime → timestamp 注意时区
4. 切换应用配置后重启
